## 二、接口约定文档

### 2.1 通用约定

**Base URL**: `http://localhost:8080/api`

**统一响应格式**：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

**HTTP 状态码约定**：

| code | 含义 |
|:---|:---|
| 200 | 请求成功 |
| 400 | 参数错误（如必填字段为空、格式非法） |
| 401 | 未登录或登录过期 |
| 403 | 权限不足（如患者访问管理员接口） |
| 404 | 资源不存在 |
| 409 | 业务冲突（如号源已约满、重复挂号） |
| 500 | 服务器内部错误 |

**登录验证方式**：请求头携带 `token`（登录接口返回），后端通过拦截器校验。

## 三、协作注意事项

### 3.1 Git 分支管理

```
main          ← 稳定分支，仅合并已验证的功能
feature/backend-ddc   ← 狄东琛的后端分支
feature/frontend-xgy  ← 肖国扬的前端分支
feature/database-fqm    ← 范骐鸣的 SQL/数据分支
```

- 每人**只在自己的分支上提交**，不要直接往 main 上推
- 每天开始工作前，**先 `git pull origin main`** 同步他人的进展
- 合并到 main 前，先在新分支上验证能正常运行
- 提交信息用中文，格式：`feat: 说明` / `fix: 说明` / `docs: 说明`

### 3.2 前后端联调约定

1. **字段命名**：后端统一用 camelCase（JSON），前端也用 camelCase。数据库用 snake_case（已有的设计）。MyBatis-Plus 配置 `map-underscore-to-camel-case: true` 自动转换。
2. **日期格式**：接口中日期字符串统一用 `yyyy-MM-dd`（如 `2026-05-20`），时间用 `yyyy-MM-ddTHH:mm:ss`。
3. **空值处理**：后端不返回 `null` 值的字段，或用空字符串 `""` 代替，避免前端渲染报错。
4. **错误消息**：后端返回的错误 `message` 字段内容直接展示给用户，所以消息要写中文且清晰（如"该排班号源已约满"而不是"Schedule quota exhausted"）。
5. **联调顺序**：先调通科室查询（最简单），再调挂号（最关键），最后调管理端。

### 3.3 关键业务规则

| 规则 | 实现位置 | 说明 |
|:---|:---|:---|
| 挂号时号源 -1 | 后端 Service `@Transactional` | 两操作（插入 Appointment + 更新 rest_quota）在同一事务中 |
| 取消时号源 +1 | 后端 Service `@Transactional` | 同上，事务保证一致性 |
| 号源不能 < 0 | 数据库 CHECK 约束 + 后端判断 | 双重保障，防超卖 |
| 同一身份证只能注册一次 | 数据库 UNIQUE + 后端判断 | UNIQUE 是最后防线 |
| 同医生同时段不能重复排班 | 数据库复合 UNIQUE | `(doc_id, work_date, shift)` |
| 有患者已挂号时不能删除排班 | 数据库 FK RESTRICT | 后端捕获异常返回 409 |
| 有医生时不能删除科室 | 数据库 FK RESTRICT | 同上 |
| 一个患者不能对同一排班重复挂号 | 后端业务判断 + 数据库 UNIQUE 索引 | 建议在 `Appointment` 表加 `(patient_id, schedule_id)` 唯一约束且 status=1 时生效 |

### 3.4 数据初始化约定

数据库负责人需要保证测试数据**覆盖以下场景**：

| 场景 | 数据示例 |
|:---|:---|
| 有号源的排班 | rest_quota > 0 |
| 已约满的排班 | rest_quota = 0 |
| 不同科室、不同医生 | 至少 3 个科室，每科室 2-3 个医生 |
| 不同状态的挂号记录 | 已预约 / 已取消 / 已完成 各若干条 |
| 同一患者有多条预约 | 用于测试"我的预约"列表 |
