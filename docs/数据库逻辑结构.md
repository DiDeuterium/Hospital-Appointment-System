## 二、 主要约束设计说明

本数据库设计严格遵循将 **“业务规则下沉到数据库层”** 的原则，通过各类约束保障数据的完整性与合法性，具体设计思路如下：

1.  **实体完整性 (主键 PRIMARY KEY 约束)**
    *   **业务体现：** 所有表均设置了单字段主键。`Department`、`Doctor`、`Patient`、`Schedule`、`Appointment` 五张表的主键全部采用 `INT AUTO_INCREMENT` 代理主键，INT 类型在多表关联中的查询效率远高于字符串，同时也规避了身份证号等敏感信息泄露的风险。
2.  **参照完整性 (外键 FOREIGN KEY 约束)**
    *   **业务体现：** `Appointment` 表中的 `patient_id` 和 `schedule_id` 必须引用实际存在的患者和排班记录。我们采用 `ON DELETE RESTRICT` 策略：如果一个排班已经被患者预约，则系统在数据库层面拒绝管理员删除该排班的操作，防止产生“幽灵预约单”。
3.  **非空约束 (NOT NULL)**
    *   **业务体现：** 关键业务数据如医生的所属科室 (`dept_id`)、排班的日期 (`work_date`)、患者的手机号 (`phone`) 等均设置为非空，杜绝脏数据录入导致后续系统报错。
4.  **唯一约束 (UNIQUE)**
    *   **业务体现：**
        *   `Patient.id_card` 设为唯一，防止同一个患者使用身份证重复注册。
        *   `Department.dept_name` 设为唯一，防止管理员手误录入两个同名科室。
        *   在 `Schedule` 表中，我们将 `(doc_id, work_date, shift)` 设为复合唯一约束，从数据库物理底层保证一名医生在同一天的同一个时间段只能存在一个排班记录。
5.  **检查约束 (CHECK)**
    *   **业务体现 (防超卖/防黄牛)：** 为 `Schedule.rest_quota` 字段添加 `CHECK (rest_quota >= 0)` 约束。在后续并发挂号场景中，即使高并发导致代码层的判断失效，数据库底层的 CHECK 约束也会强制阻断剩余号源变成负数（即防超卖），保障核心业务的绝对安全。
    *   **业务体现 (数据规范)：** 约束性别字段 `gender IN ('M', 'F')`，状态字段 `status IN (1, 2, 3)`。
6.  **默认值 (DEFAULT)**
    *   **业务体现：** `Appointment.create_time` 默认值为 `CURRENT_TIMESTAMP`，让数据库自动记录挂号发生的精确时间。`status` 默认为 `1`（已预约），简化前端插入数据的 SQL 语句。

## 三、 核心 SQL 草稿 (DDL)

```sql
-- 1. 创建科室表
CREATE TABLE Department (
    dept_id INT AUTO_INCREMENT PRIMARY KEY,
    dept_name VARCHAR(50) NOT NULL UNIQUE,
    location VARCHAR(100),
    description TEXT
);

-- 2. 创建医生表
CREATE TABLE Doctor (
    doc_id INT AUTO_INCREMENT PRIMARY KEY,
    doc_name VARCHAR(50) NOT NULL,
    gender CHAR(1) NOT NULL CHECK (gender IN ('M', 'F')),
    title VARCHAR(20),
    dept_id INT NOT NULL,
    password VARCHAR(255) NOT NULL,
    FOREIGN KEY (dept_id) REFERENCES Department(dept_id) ON DELETE RESTRICT
);

-- 3. 创建患者表
CREATE TABLE Patient (
    patient_id INT AUTO_INCREMENT PRIMARY KEY,
    id_card CHAR(18) NOT NULL UNIQUE,
    real_name VARCHAR(50) NOT NULL,
    gender CHAR(1) CHECK (gender IN ('M', 'F')),
    phone VARCHAR(20) NOT NULL,
    password VARCHAR(255) NOT NULL
);

-- 4. 创建排班表
CREATE TABLE Schedule (
    schedule_id INT AUTO_INCREMENT PRIMARY KEY,
    doc_id INT NOT NULL,
    work_date DATE NOT NULL,
    shift VARCHAR(10) NOT NULL CHECK (shift IN ('上午', '下午', '夜诊')),
    total_quota INT NOT NULL CHECK (total_quota > 0),
    rest_quota INT NOT NULL,
    -- 复合唯一约束：确保同个医生同一天同时段只有一个排班
    UNIQUE (doc_id, work_date, shift), 
    -- 检查约束：防止号源超卖
    CHECK (rest_quota >= 0 AND rest_quota <= total_quota), 
    FOREIGN KEY (doc_id) REFERENCES Doctor(doc_id) ON DELETE RESTRICT
);

-- 5. 创建挂号记录表 (多对多关联表)
CREATE TABLE Appointment (
    appt_id INT AUTO_INCREMENT PRIMARY KEY,
    patient_id INT NOT NULL,
    schedule_id INT NOT NULL,
    status INT NOT NULL DEFAULT 1 CHECK (status IN (1, 2, 3)), -- 1:已预约 2:已取消 3:已完成
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (patient_id) REFERENCES Patient(patient_id) ON DELETE RESTRICT,
    FOREIGN KEY (schedule_id) REFERENCES Schedule(schedule_id) ON DELETE RESTRICT
);
```

## 四、 索引方案设计说明

> 索引创建语句已整合到 `sql/schema.sql` 中，建表后自动执行。本节说明每条索引的设计依据。

### 4.1 索引清单

| 索引名 | 所属表 | 索引字段 | 索引类型 | 设计理由 |
| :--- | :--- | :--- | :--- | :--- |
| `PRIMARY` | department | dept_id | 主键索引(自增) | 实体完整性, 科室唯一标识 |
| `PRIMARY` | doctor | doc_id | 主键索引(自增) | 实体完整性, 医生唯一标识 |
| `idx_doctor_dept_id` | doctor | dept_id | 普通索引 | 高频联表: 查某科室下所有医生(患者级联浏览第二步) |
| `PRIMARY` | patient | patient_id | 主键索引(自增) | 实体完整性, 代理主键 |
| `id_card` | patient | id_card | 唯一索引 | 业务唯一标识, 同时加速登录时的身份证查询 |
| `PRIMARY` | schedule | schedule_id | 主键索引(自增) | 实体完整性 |
| `idx_schedule_doc_id` | schedule | doc_id | 普通索引 | 联表: 查医生的排班列表(医生端"我的排班") |
| `idx_schedule_date` | schedule | work_date | 普通索引 | 范围查询: 按日期筛选排班(患者端"未来7天号源") |
| `doc_id_work_date_shift` | schedule | (doc_id, work_date, shift) | 复合唯一索引 | 同时加速 "某医生某天某时段" 的精确查找 |
| `PRIMARY` | appointment | appt_id | 主键索引(自增) | 实体完整性 |
| `idx_appt_patient` | appointment | patient_id | 普通索引 | 联表: 查某患者的所有预约("我的预约"列表) |
| `idx_appt_schedule` | appointment | schedule_id | 普通索引 | 联表: 查某排班下的所有患者(医生端"患者名册") |
| `idx_appt_status` | appointment | status | 普通索引 | 条件筛选: 按状态过滤(已预约/已取消/已完成) |

### 4.2 高频查询路径与索引命中分析

以下对照 `queries.sql` 中的 7 个核心查询，逐条验证索引覆盖情况：

| 查询 | SQL 核心条件 | 命中索引 | 扫描类型 |
| :--- | :--- | :--- | :--- |
| Q1 科室列表 | 全表扫描 department | `PRIMARY` (隐含排序) | 全表 (表极小, 无需额外索引) |
| Q2 科室下的医生 | `WHERE dept_id = ?` | `idx_doctor_dept_id` | **ref** (精准命中) |
| Q3 患者信息 | `WHERE patient_id = ?` 或 `WHERE id_card = ?` | `PRIMARY` 或 `id_card` UNIQUE | **const** (唯一命中) |
| Q4 排班+医生+科室 | `WHERE dept_id=? AND work_date=?` | `idx_schedule_date` + `idx_doctor_dept_id` | **ref** (两张表分别精准命中) |
| Q5 患者预约列表 | `WHERE a.patient_id = ?` | `idx_appt_patient` | **ref** (精准命中) |
| Q6 医生预约统计 | `GROUP BY d.doc_id` + 联表 | 各表 PRIMARY | 全表聚合 (统计类查询, 无筛选条件) |
| Q7 未来7天有号源科室 | `WHERE work_date BETWEEN ? AND ? AND rest_quota > 0` | `idx_schedule_date` | **range** (范围命中) |

### 4.3 索引设计原则总结

1. **高频筛选字段优先建索引**: `dept_id`(科室筛选)、`work_date`(日期筛选)、`status`(状态筛选) 是用户端最高频的查询条件。
2. **外键默认建索引**: `doctor.dept_id`、`schedule.doc_id`、`appointment.patient_id`、`appointment.schedule_id` 均为外键，联表查询频繁，必须建索引避免全表扫描。
3. **复合唯一约束自带索引**: Schedule 表的 `(doc_id, work_date, shift)` 唯一约束本身就是一个复合索引，同时满足约束需求和查询加速。
4. **小表不建多余索引**: department 表数据量极小(通常 < 100 行)，全表扫描成本可忽略，无需额外建索引。
5. **未对 `rest_quota` 建索引**: 虽然 Q7 中有 `rest_quota > 0` 条件，但该字段区分度低(仅有 "=0" 和 ">0" 两种值)，不适合建独立索引，实际查询中 `work_date` 的范围索引已大幅缩小扫描范围。
6. **关于去重索引的说明**: MySQL 8.0 不支持条件唯一索引，因此 `Appointment(patient_id, schedule_id)` 不能使用 `WHERE status = 1` 的部分唯一约束。重复挂号的最终防线由后端业务代码实现(SELECT COUNT 判断)，数据库层的 `UNIQUE` 约束无法区分状态。
